# 프론트엔드 작업 프롬프트

## 프로젝트 개요
- 이 폴더는 Next.js(App Router) + React + TypeScript 기반 프론트엔드다.
- 주요 사용자 페이지는 `app/egogift`, `app/cardpack`, `app/event`, `app/favorites`에 위치한다.
- 관리자 페이지는 `app/dante/(admin)` 하위에 있으며, 생성/수정/매핑 작업 화면이 집중되어 있다.

## 기술 및 실행 정보
- 패키지 매니저: `npm` (`package-lock.json` 기준)
- 주요 스크립트:
  - 개발: `npm run dev`
  - 빌드: `npm run build`
  - 린트: `npm run lint`
- API 기본 주소는 `lib/api.ts`에서 `NEXT_PUBLIC_API_BASE_URL` 또는 `http://localhost:8080/api`를 사용한다.

## 코드 구조 가이드
- 라우팅/페이지: `app/**/page.tsx`, 공통 레이아웃: `app/**/layout.tsx`
- 공용 UI 컴포넌트: `components/*`
- 공용 유틸/상수/API 클라이언트: `lib/*`
- 서버 라우트(프록시 등): `app/api/*`

## 작업 원칙
- App Router 패턴(서버/클라이언트 컴포넌트 경계)을 유지하고, 불필요한 `use client`를 추가하지 않는다.
- 기존 `lib/api.ts` axios 인스턴스를 우선 사용해 API 호출 일관성을 유지한다.
- 관리자(`app/dante/(admin)`)와 사용자 페이지의 동작/권한 흐름을 섞지 않는다.
- 변경 시 영향 범위가 큰 화면(`egogift`, `cardpack`, `favorites`)은 상태/필터 파라미터 유지 여부를 먼저 확인한다.
- 스타일은 현재 프로젝트의 전역 스타일(`app/globals.css`)과 기존 컴포넌트 패턴을 우선 따른다.

## 작업 체크리스트
1. 변경 대상 라우트/컴포넌트의 기존 데이터 흐름 확인
2. 타입 정의와 API 응답 형태 일치 여부 검증
3. 최소 범위 수정 후 `npm run lint`로 정적 점검
4. 사용자 페이지 + 관리자 페이지 회귀 영향 확인

## 응답/작업 방식
- 이후 이 폴더 관련 작업은 이 문서를 기본 컨텍스트로 삼아 구현한다.
- 새로운 규칙이 생기면 이 문서에 먼저 반영한 뒤 작업을 진행한다.
